#NijuueiRemake
## 怎么运行
(AI)生成

```bash
cd onscripter_project
./onscripter-en --window     # 窗口模式
./onscripter-en               # 全屏
```

这个 `onscripter-en` 是在本机（Linux x86-64）用
[Galladite27/ONScripter-EN](https://github.com/Galladite27/ONScripter-EN) 源码编译的二进制。
如果你要在 Windows 上玩，需要参照该仓库的 `BuildInstructions.md` 在 Windows/MSYS2 下另外编译一份，
或者用任何支持 UTF-8 模式（文件名 `0.utf`）的较新 ONScripter-EN 官方发行版直接把这个文件夹整个丢进去运行。

首次运行需要窗口里能看到"「二重影」ONScripter 重制版"标题字样和四个选项：
`开始正篇 / 番外篇 / 章节选择 / 退出`。"章节选择"可以直接输入章节号(1-46 或 101-118)跳转，方便你抽查任意一章。

## 目录结构

```
onscripter_project/
  0.utf              最终生成的游戏脚本（UTF-8模式，纯文本，可以直接打开看）
  default.ttc         字体(系统自带 Noto Sans CJK SC，简体中文覆盖完整)
  onscripter-en       引擎可执行文件(Linux)
  bg/ fg/ bgm/ se/ voice/   从 ons_game 拷贝过来的素材(原文件名/路径基本不变)
  tools/              生成 0.utf 的整套 Python 流水线，改完 extracted_text 或调整规则后重新跑一遍即可
    parse_records.py   Stage1: 把 extracted_text/*.txt 切成"已分类的token"
    build_scenes.py    Stage2: 归属说话人、解析背景/立绘/语音引用、切分选择支和到访变体
    gen_onscript.py    Stage3: 把结构化场景转成 ONScripter 脚本文本
    build_all.py       总装：跑1+2+3，拼标题/章节选择/章节链，写0.utf，拷贝素材
    charmap.py         角色名 -> 颜色 的对照表(可以在这里改颜色、改别名归并)
    validate.py         对生成的0.utf做静态校验(标签/文件引用/引号配对)
    ir/                中间产物(每章一个 json，方便你直接查看某一章解析结果)

重新生成整个游戏，只需要:
    cd tools && python3 build_all.py && python3 validate.py
```

## 这一版做了什么、没做什么（请务必看一下）

这是从"文本抽取结果"反推回一个可运行游戏，抽取结果本身不是完整的原始脚本（没有分支跳转的
字节码，只有走过的那条线性文字记录），所以有一些地方是"尽力还原"而不是"精确还原"：

1. **说话人归属是启发式的**：能明确识别的是"人名单独一行紧跟语音/对话"和"人名+全角空格+「对话」"
   这两种原文里出现的写法；除此之外，对话没有具名时按"沿用上一个已知说话人，遇到旁白就重置为
   无名"处理。大部分场景下这个规则是对的，但快速你来我往的对话可能会被错误地都算成同一个人。
   角色名->颜色对照表在 `tools/charmap.py`，已知主要角色（双厳/楓玲/志乃/桔梗/紗枝/伊魯水/北条/
   幽香/風喜/桂華/十兵衛/彩/無影/老人）给了固定颜色，没收录的名字会自动按名字哈希出一个颜色，
   不会报错也不会漏内容。

2. **探索选择支（"◆移動到XX"这类）**：原始抽取结果本质上是走过的单一路径，同一个选择点里，
   没被走过的选项文本在语料里完全找不到。处理方式：
   - 菜单正常列出当年所有选项文字（不会阉割）
   - 有内容的选项都指向抽取到的那份内容（换句话说，选哪个都会看到"抓取到的那条"），这是你确认过
     的"缺失分支路由到可用分支"方案的直接推广
   - 同一地点反复进入、文案变短的效果**完整保留**了——用一个全局计数变量记录到访次数，第1次/第2次/
     第3次...分别显示原文里对应的那一段，超出变体数就一直显示最后一段
   - 每个选择菜单都会自动多加一个"▶ 继续故事"选项，用来跳出循环、把主线剧情往下推——这是原文里
     没有的，是为了保证"探索环节一定能走出去"而加的兜底项，界面上会显得比正篇选项更显眼一些
   - 只有 1 个选项的"分支"（通常是"回到XX"这种过场提示）不会做成选择菜单，直接当过场字幕+顺序
     播放内容，因为只有一个选项谈不上"选择"

3. **背景音乐**：抽取的文本里完全没有任何 BGM 切入点的记录（背景/立绘用的是同一套 `Kgd\` 指令，
   BGM 大概率写在更上层、没有做文本抽取的 Kas 文件里）。目前的处理是**按章节顺序轮流播放 bgm 目录
   里的22条曲子**，和原版的实际配乐时间点对不上，只是保证"一直有音乐在放"。如果你想精确配乐，
   需要人工按剧情逐章重新指定，可以改 `tools/build_all.py` 里的 `bgm_for_index`。

4. **文本本身没有做任何修改**（按你的要求，这一轮只搭引擎和流程）。语料里已知有两类问题还没处理：
   - 约17%的行混有"堯/龜/槇/髪/遙"等固定误码（汉化组当年"偷字"导致，规律稳定但没修）
   - 约50多处台词还是没翻完的日文原文
   这些以后要修的话，直接改 `ons_game/extracted_text/*.txt` 原文，再重新跑一遍
   `tools/build_all.py` 就会自动带入新脚本，不需要碰生成器代码。

5. **立绘/背景的显示方式是按图片实际尺寸自动判断的**：800x600 的算"通栏大图"（背景/事件CG，
   用 `bg` 指令铺满屏幕）；高度=600但宽度<800 的算"人物差分"（用 `ld c` 居中显示，比如
   彩/北条/十兵衛这些角色的立绘）。这个判断是看图片像素尺寸得出的，不是看文件夹，所以理论上应该
   比较准，但没有把每一张图都人工核对过。

6. **语音只在有对应`Vox\`引用的地方播放**，走 `dwave` 声道0；音效走声道1；BGM走声道2循环播放。

7. **没有做存档界面美化、回想模式、CG鉴赏等系统菜单定制**，用的是引擎默认右键菜单
   （因为 `language japanese`，默认菜单文字是日文——这是为了让中文这种没有空格的文本能按字正确
   换行选的，如果想要中文/英文菜单文字，需要另外用 `rmenu`/`savename` 等指令自定义，这个目前没做）。

8. 我没能在这台机器上跑出可视化截图确认字体/排版效果（这台机器是 Wayland 桌面，没有装 Xvfb/grim
   这类截图工具，也没有 sudo 权限安装）；已经确认脚本本身能被引擎正确加载、解析（用 `validate.py`
   做了标签/文件引用/引号配对的全量静态检查，全部通过），但实际显示效果、字号/行距是否合适、
   文本框位置是否得当，需要你自己跑起来看一眼，`setwindow4` 那一行（在 `0.utf` 第18行）可以直接
   手改数字微调。

   BGM 大概率写在更上层、没有做文本抽取的 Kas 文件里）。目前的处理是**按章节顺序轮流播放 bgm 目录
   里的22条曲子**，和原版的实际配乐时间点对不上，只是保证"一直有音乐在放"。如果你想精确配乐，
   需要人工按剧情逐章重新指定，可以改 `tools/build_all.py` 里的 `bgm_for_index`。

4. **文本本身没有做任何修改**（按你的要求，这一轮只搭引擎和流程）。语料里已知有两类问题还没处理：
   - 约17%的行混有"堯/龜/槇/髪/遙"等固定误码（汉化组当年"偷字"导致，规律稳定但没修）
   - 约50多处台词还是没翻完的日文原文
   这些以后要修的话，直接改 `ons_game/extracted_text/*.txt` 原文，再重新跑一遍
   `tools/build_all.py` 就会自动带入新脚本，不需要碰生成器代码。

5. **立绘/背景的显示方式是按图片实际尺寸自动判断的**：800x600 的算"通栏大图"（背景/事件CG，
   用 `bg` 指令铺满屏幕）；高度=600但宽度<800 的算"人物差分"（用 `ld c` 居中显示，比如
   彩/北条/十兵衛这些角色的立绘）。这个判断是看图片像素尺寸得出的，不是看文件夹，所以理论上应该
   比较准，但没有把每一张图都人工核对过。

6. **语音只在有对应`Vox\`引用的地方播放**，走 `dwave` 声道0；音效走声道1；BGM走声道2循环播放。

7. **没有做存档界面美化、回想模式、CG鉴赏等系统菜单定制**，用的是引擎默认右键菜单
   （因为 `language japanese`，默认菜单文字是日文——这是为了让中文这种没有空格的文本能按字正确
   换行选的，如果想要中文/英文菜单文字，需要另外用 `rmenu`/`savename` 等指令自定义，这个目前没做）。

8. 我没能在这台机器上跑出可视化截图确认字体/排版效果（这台机器是 Wayland 桌面，没有装 Xvfb/grim
   这类截图工具，也没有 sudo 权限安装）；已经确认脚本本身能被引擎正确加载、解析（用 `validate.py`
   做了标签/文件引用/引号配对的全量静态检查，全部通过），但实际显示效果、字号/行距是否合适、
   文本框位置是否得当，需要你自己跑起来看一眼，`setwindow4` 那一行（在 `0.utf` 第18行）可以直接
   手改数字微调。
